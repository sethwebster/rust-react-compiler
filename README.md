# rust-react-compiler

A Rust port of Meta's React compiler — the same compiler that ships in React 19 to automatically memoize components and hooks. The TypeScript original is ~56,000 lines across 55 sequential passes. This project ports that pipeline pass-by-pass to native Rust using [oxc](https://github.com/oxc-project/oxc) for parsing.

**Status: active development.** Phase 1 (parsing, HIR lowering, SSA, pre-lowering validation) is complete. Phase 2 (reactive scope inference, dependency propagation) is ~35% done. Phase 3 (build_reactive_function, scope-aware codegen) is ~5% done.

---

## Why

The TypeScript React compiler requires Node.js, runs as a Babel plugin, and carries the runtime overhead of the JS engine. A native Rust binary can:

- Parse and compile files with no Node.js dependency
- Run significantly faster on large codebases (Rust vs. interpreted TypeScript)
- Be embedded in non-Node.js build toolchains (Rust bundlers, Go tools, etc.)
- Serve as a reference implementation for future Rust-based React tooling

The long-term goal is a `react-compiler` binary that produces the same output as the Meta compiler with zero Node.js dependency.

## Architecture

The compiler is a sequential pipeline. Each pass takes the HIR (High-level Intermediate Representation) and returns a transformed HIR, with the final pass producing JavaScript source text.

```
oxc parse (0.69)
    |
    v
pre-lowering validators         Static analysis before HIR construction.
    |                           Catches: optional-dep mismatches, indirect
    v                           props mutations in effects, rule violations.

HIR lowering                    oxc AST → HIRFunction. Constructs basic
    |                           blocks, Places, and Instructions.
    v

SSA construction                Renames variables to SSA form.
    |                           Inserts phi nodes at join points.
    v

Type inference                  Infers React-specific types: hook, component,
    |                           primitive, object, unknown.
    v

Inference passes                Computes mutable live ranges, reactive places,
    |                           aliasing effects.
    v

Optimization passes             Dead code elimination, function outlining,
    |                           constant propagation.
    v

Reactive scope inference        Groups instructions into reactive scopes —
    |                           the regions that will become useMemo/useCallback.
    v

Reactive scope transforms       Merges, prunes, and propagates dependencies
    |                           across scope boundaries.
    v

build_reactive_function         HIRFunction → ReactiveFunction.       [PARTIAL]
    |                           This is the critical missing piece.
    v

Codegen                         Emits JavaScript. Currently operates on
    |                           HIR directly (architectural mismatch that
    v                           will be resolved when ReactiveFunction exists).

oxc_codegen → JS output
```

### HIR Design

The HIR is a flat, block-based intermediate representation inspired by the TypeScript compiler's HIR but adapted for Rust's ownership model.

- **IDs are `u32` newtypes** — `IdentifierId`, `BlockId`, `InstructionId`, `ScopeId`, `DeclarationId`. All generated by the `opaque_id!` macro. No pointer aliasing.
- **`Place`** stores an `IdentifierId`, not a reference. Identifier data lives in `Environment.identifiers: HashMap<IdentifierId, Identifier>`. Look up with `env.identifier(id)`.
- **`HIRFunction.blocks`** is `IndexMap<BlockId, BasicBlock>` in reverse-postorder. Iterating the map gives you RPO traversal for free.
- **No lifetimes on HIR types** — all owned `String`s. This avoids borrow checker complexity at the cost of some cloning, which is acceptable at this stage.
- **`serde::Serialize/Deserialize`** on all HIR types.

### Parser

This project uses [oxc](https://github.com/oxc-project/oxc) 0.69, not Babel. oxc parses JS, TS, JSX, and TSX natively. The AST node shapes differ from Babel — do not assume field name parity when referencing the TypeScript compiler's lowering code.

## Build

Prerequisites: Rust stable (1.77 or later).

```bash
# Clone (note: react/ is a submodule — do not initialize it)
git clone https://github.com/sethwebster/rust-react-compiler
cd rust-react-compiler

# Build the compiler binary
cd rust-react-compiler
cargo build --release

# Run on a single file
cargo run -- path/to/component.jsx
```

The binary accepts a single file path and writes the compiled JavaScript to stdout.

## Tests

The test suite runs the compiler against 1,244 fixtures from the TypeScript React compiler's own test suite and measures both compile rate and correct rate.

```bash
cd rust-react-compiler

# Run all fixtures (takes ~30–60s)
cargo test --test fixtures run_all_fixtures -- --ignored

# Show summary metrics only
cargo test --test fixtures run_all_fixtures -- --ignored 2>&1 | grep -E "Compile rate|Correct rate|Error"

# Show per-fixture output
cargo test --test fixtures run_all_fixtures -- --ignored --nocapture 2>&1 | tail -100
```

### How Fixtures Work

Each fixture is a JS/TS/JSX/TSX file. For non-error fixtures, a companion `.expect.md` contains the expected compiled output in a fenced `## Code` section. The test harness:

1. Compiles the fixture with our Rust compiler.
2. Extracts the expected JS from the `.expect.md`.
3. Normalizes both outputs (collapses whitespace, strips comments and trailing commas).
4. Compares them for equality.

Fixtures named `error.*` or `todo.error.*` are expected to fail compilation. If our compiler accepts one of these, it is counted as an `error_unexpected` — a hard regression.

## Critical Path

The correct rate is currently 24.1%. The primary bottleneck is `build_reactive_function`.

The TypeScript compiler transforms a post-scope-inference `HIRFunction` into a `ReactiveFunction` — a tree of reactive scopes that directly maps to `useMemo`/`useCallback` call sites. Our codegen currently bypasses this and operates on raw HIR, which means it cannot emit the scope-based memoization wrapping that most fixtures expect.

The three steps that will unlock significant correct rate improvement:

1. **Define `ReactiveFunction` in `hir/hir.rs`** — model after `ReactiveScopes/BuildReactiveFunction.ts` and `ReactiveScopes/ReactiveFunction.ts` in the TypeScript source.
2. **Implement `build_reactive_function`** — convert `HIRFunction` (post-scope inference) to `ReactiveFunction`.
3. **Rewrite `codegen_reactive_function`** to operate on `ReactiveFunction` and emit `useMemo`/`useCallback` wrapping.

## Development Workflow

### Branch naming

Branches should be prefixed with your identifier, e.g. `seth/build-reactive-function`.

### Commit messages

Follow conventional commits: `feat`, `fix`, `perf`, `refactor`, `test`, `docs`, `chore`. Keep subjects under 50 characters, imperative mood.

### Before committing

```bash
cd rust-react-compiler
cargo check                    # Fast type check
cargo test --test fixtures run_all_fixtures -- --ignored 2>&1 | grep -E "Compile rate|Correct rate|Error"
```

Do not commit changes that reduce compile rate, reduce correct rate, or introduce any `error_unexpected` results.

### Agent sessions

If you are an AI agent: read `AGENT-STATE.md` before starting and update it when you stop. The `AGENT-WORKSPACE.md` file has the full session protocol, directory layout, pipeline order with per-pass status, architecture decisions, and known infrastructure gotchas.
